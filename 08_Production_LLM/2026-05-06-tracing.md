# Daily Review Note: Tracing

## 1. Core idea

**Tracing** — це спосіб побачити повний шлях виконання одного запиту або операції через систему.

Коротко:

```text
Tracing = запис усіх кроків, які пройшов один request / operation
```

У backend tracing відповідає на питання:

> Що саме сталося з цим конкретним запитом від початку до кінця?

---

## 2. Why it matters

У distributed systems один запит часто проходить через багато компонентів:

```text
API Gateway
  ↓
Backend Service
  ↓
Database
  ↓
Kafka
  ↓
External API
  ↓
Response
```

Без tracing важко зрозуміти:

- де саме запит сповільнився;
- який сервіс повернув помилку;
- який external call дав timeout;
- який DB query був дорогий;
- який крок chain / agent / RAG pipeline зламався.

Для LLM-систем tracing ще важливіший, бо LLM flow часто є black box:

```text
user input
  ↓
prompt template
  ↓
retrieval
  ↓
context injection
  ↓
LLM call
  ↓
tool call
  ↓
output parser
  ↓
final answer
```

---

## 3. How it works

Tracing складається з трьох базових понять:

### Trace

**Trace** — це повна історія одного request / operation.

```text
Trace = вся execution tree одного запиту
```

Приклад:

```text
Trace: POST /api/payments
```

---

### Span

**Span** — це один крок всередині trace.

Приклади span:

```text
HTTP request to user-service
SQL query
Kafka publish
LLM call
Vector DB search
Tool call
Output parsing
```

Кожен span зазвичай має:

- name;
- start time;
- end time;
- duration;
- status;
- attributes / metadata;
- error, якщо він стався.

---

### Trace ID

**Trace ID** — унікальний ідентифікатор одного запиту.

```text
traceId=7f3a9c2...
```

Він дозволяє звʼязати всі spans одного request між різними сервісами.

---

## 4. Backend analogy

Уяви запит:

```text
POST /api/payments
```

Він проходить через:

```text
API Gateway
  ↓
Payment Service
  ↓
User Service
  ↓
Kafka
  ↓
Fraud Service
  ↓
PostgreSQL
  ↓
External Payment Provider
```

Tracing покаже execution path одного конкретного запиту:

```text
traceId=abc-123

API Gateway                 20 ms
Payment Service             80 ms
PostgreSQL query            40 ms
Kafka publish               15 ms
External Provider call      900 ms  ← bottleneck
```

Висновок:

```text
Проблема не в gateway і не в database.
Основний bottleneck — external payment provider.
```

---

## 5. Production relevance

### Logs vs Metrics vs Traces

| Інструмент | Що показує | Приклад |
|---|---|---|
| Logs | Що сталося в конкретному місці | `Failed to call OpenAI` |
| Metrics | Агреговані числа | `p95 latency = 2.3s` |
| Traces | Повний шлях одного запиту | `Gateway → Service → LLM → Tool → Response` |

Production observability зазвичай потребує всі три:

```text
logs + metrics + traces = observability
```

---

### Tracing у LangChain / LangSmith

Для LangChain коду:

```python
chain = prompt_template | llm
response = chain.invoke({"information": information})
```

LangSmith може показати:

```text
RunnableSequence
  ↓
PromptTemplate
  ↓
ChatOpenAI
  ↓
AIMessage
```

Тобто видно:

- який input прийшов;
- який prompt реально був згенерований;
- яка model була викликана;
- яка відповідь прийшла;
- скільки це зайняло часу;
- скільки tokens витрачено;
- чи була помилка;
- який step chain зламався.

---

### Чому tracing критичний для LLM apps

Без tracing ти часто не знаєш:

- який prompt реально пішов у модель;
- чи RAG знайшов правильні документи;
- чому agent вибрав неправильний tool;
- чи output parser зламався;
- скільки коштував запит;
- де виникла latency;
- чи модель hallucinated через поганий context.

Особливо важливо для:

- chains;
- agents;
- tools;
- RAG;
- vector DB;
- external APIs;
- multiple LLM providers;
- retries;
- fallbacks;
- streaming;
- structured output.

## 7. Common mistakes

### Mistake 1 — Плутати logs і traces

Logs — це окремі записи подій.

Tracing — це повний шлях одного запиту.

```text
Logs answer: what happened here?
Tracing answers: what happened end-to-end?
```

---

### Mistake 2 — Не прокидати traceId між сервісами

Якщо traceId не передається між gateway, services, Kafka consumers і external calls, trace буде розірваним.

Production-ризик:

```text
Неможливо побачити повний request path.
```

---

### Mistake 3 — Не trace-ити external calls

Часто bottleneck не в твоєму коді, а в:

- external API;
- database;
- vector DB;
- LLM provider;
- payment provider;
- email/SMS provider.

---

### Mistake 4 — Не trace-ити LLM prompts

У LLM apps критично бачити final prompt, бо саме він визначає поведінку моделі.

Без цього debugging буде слабким.

---

### Mistake 5 — Ігнорувати token/cost metadata

Для LLM production важливо trace-ити:

- input tokens;
- output tokens;
- model name;
- latency;
- retries;
- failed calls;
- cost per request.

## 9. Interview Q&A

### 1. What is tracing?

Tracing is the ability to observe the end-to-end execution path of a single request across components, services, databases, queues, external APIs, or LLM calls.

---

### 2. What is the difference between logs, metrics, and traces?

Logs describe individual events, metrics provide aggregated numerical signals, and traces show the complete path of one request through the system.

---

### 3. What is a span?

A span is a single operation within a trace, such as an HTTP call, SQL query, Kafka publish, vector search, or LLM call.

---

### 4. Why is tracing important in distributed systems?

Because one request may cross many services. Tracing helps identify latency bottlenecks, failed dependencies, broken propagation, and the exact component responsible for an issue.

---

### 5. Why is tracing important in LLM applications?

Because LLM applications often involve prompts, model calls, retrieval, tools, agents, parsers, and retries. Tracing makes these hidden steps visible and debuggable.

---

### 6. What should be traced in a production LLM app?

User input, rendered prompt, retrieved documents, model call, token usage, latency, tool calls, parser output, retries, fallback model usage, and final response.

## Production takeaway

Production-система без tracing — це частково blind system.

Production LLM-система без tracing — це майже black box.

Staff-level висновок:

```text
Tracing не просто допомагає дебажити.
Tracing робить поведінку distributed і LLM-систем спостережуваною, вимірюваною і керованою.
```
